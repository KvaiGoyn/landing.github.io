# Архитектура проекта MetalPro

## Обзор архитектуры

Проект MetalPro построен на основе Next.js 16 с использованием App Router и следует принципам модульной, масштабируемой архитектуры. Основная цель — разделение ответственности, улучшение поддерживаемости и подготовка к будущему расширению.

## Принципы проектирования

### 1. Принцип единой ответственности (SRP)
Каждый компонент, хук или контекст отвечает за одну конкретную задачу:
- `Button` — только отображение и взаимодействие с кнопкой
- `ModalContext` — только управление модальными окнами
- `useForm` — только управление состоянием формы

### 2. Разделение состояний
Вместо монолитного `AppContext` используется несколько специализированных контекстов:
- **ModalContext**: Управление модальными окнами
- **FormContext**: Состояние форм с использованием useReducer
- **FilterContext**: Фильтры портфолио
- **UIContext**: UI состояние (мобильное меню)

### 3. Композиция над наследованием
Компоненты строятся через композицию мелких, переиспользуемых частей:
- `Button` использует `ButtonIcons` и `useButtonClasses`
- Формы используют общий хук `useForm`

### 4. Типобезопасность
Полная TypeScript типизация с:
- Общими типами в `app/types/`
- Строгими интерфейсами для пропсов
- Утилитарными типами (`Nullable<T>`, `Optional<T>`)

## Структура состояния

### Иерархия контекстов
```
AppProviders
├── ModalProvider (управление модалками)
├── FormProvider (состояние форм)
├── FilterProvider (фильтры портфолио)
└── UIProvider (UI состояние)
```

### Состояние форм (FormContext)
Использует паттерн reducer для предсказуемых обновлений:
```typescript
type FormAction =
  | { type: 'UPDATE_FORM'; payload: { formId: string; updates: Partial<FormState> } }
  | { type: 'RESET_FORM'; payload: { formId: string } }
  | { type: 'SET_SUBMITTING'; payload: { formId: string; isSubmitting: boolean } }
```

### Состояние модальных окон (ModalContext)
Простое состояние с блокировкой скролла:
```typescript
interface ModalState {
  activeModal: ModalType; // 'callback' | 'calculation' | null
  modalData: Record<string, any>;
}
```

## Компонентная архитектура

### Уровни компонентов

#### 1. UI Components (`app/components/ui/`)
Базовые, переиспользуемые компоненты:
- `Button` — с поддержкой состояний и иконок
- `Modal` — базовая модалка
- `Carousel`, `Accordion` — сложные UI компоненты

**Декомпозиция Button:**
```
Button/
├── Button.tsx           # Основной компонент
├── Button.types.ts     # Типы и константы
├── ButtonIcons.tsx     # Иконки состояний
└── useButtonClasses.ts # Логика CSS классов
```

#### 2. App Components (`app/components/`)
Компоненты приложения:
- `ModalManager` — управление всеми модалками
- Формы (`forms/`) — специализированные формы

#### 3. Page Components (`components/`)
Страничные компоненты:
- `Hero`, `Services`, `Portfolio` — секции главной страницы

### Паттерны компонентов

#### Container/Presentational
- **Presentational**: `Button`, `Modal` (только отображение)
- **Container**: `ModalManager`, формы (логика + состояние)

#### Compound Components
`Carousel` использует compound pattern:
```tsx
<Carousel>
  <Carousel.Item>...</Carousel.Item>
  <Carousel.Navigation />
</Carousel>
```

## Управление данными

### Сервисный слой (`app/services/`)
- `api.ts` — HTTP клиент с поддержкой моков
- `forms.ts` — отправка форм

### Хуки (`app/hooks/`)
Кастомные хуки для изоляции логики:
- `useForm` — управление формами с валидацией
- `useModal` — управление модалками
- `usePortfolioFilter` — фильтрация портфолио

## Маршрутизация и страницы

### App Router структура
```
app/
├── layout.tsx    # Корневой layout с провайдерами
├── page.tsx      # Главная страница
└── (future)      # Будущие маршруты
    ├── about/
    ├── portfolio/
    └── contact/
```

### Динамическая загрузка
Критические компоненты загружаются сразу, остальные — лениво:
```tsx
const ModalManager = lazy(() => import('@/app/components/ModalManager'));
```

## Производительность

### Оптимизации

#### 1. Bundle Size
- Code splitting через динамический импорт
- Tree shaking благодаря ES modules
- Оптимизированные изображения через Next.js Image

#### 2. Рендеринг
- Мемоизация через `useMemo`/`useCallback`
- Оптимизированные ре-рендеры через разделение состояния

#### 3. Загрузка
- Предзагрузка критических ресурсов
- Lazy loading для не критических компонентов
- Оптимизированные шрифты через `next/font`

### Метрики
- **LCP**: < 2.5s (оптимизированное геройское изображение)
- **FID**: < 100ms (дебаунс обработчиков)
- **CLS**: < 0.1 (стабильный layout)

## Тестирование

### Стратегия тестирования
1. **Unit тесты**: Компоненты, хуки, утилиты
2. **Integration тесты**: Взаимодействие компонентов
3. **E2E тесты**: Критические пользовательские сценарии

### Инфраструктура
- **Jest** + **React Testing Library**
- Моки для Next.js роутинга
- Настройка покрытия кода

## Масштабирование

### Добавление новых функций

#### 1. Новая страница
1. Создать маршрут в `app/`
2. Добавить компоненты в `components/`
3. Обновить навигацию

#### 2. Новый тип модалки
1. Добавить тип в `ModalType`
2. Создать компонент формы
3. Зарегистрировать в `ModalManager`

#### 3. Новый UI компонент
1. Создать в `app/components/ui/`
2. Добавить типы и тесты
3. Документировать использование

### Интеграция с бэкендом
- REST API через `ApiClient`
- WebSocket для реального времени (в будущем)
- Кэширование через React Query (рекомендуется)

## Безопасность

### Frontend безопасность
- Валидация форм на клиенте и сервере
- Защита от XSS через React
- Безопасные HTTP заголовки

### Данные пользователей
- Не храним чувствительные данные в состоянии
- Очистка данных перед отправкой
- HTTPS для всех запросов
